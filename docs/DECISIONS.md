# 아키텍처 결정 기록

스터디·포트폴리오용으로 남겨 둔 선택 요약이다.

---

## Fastify + Prisma + MySQL

- **날짜:** 6일차
- **결정:** 백엔드 스택을 Fastify + Prisma + MySQL로 둔다.
- **이유:** 라우트·플러그인 구조가 단순하고, 3일차에 잡아 둔 스키마를 Prisma로 바로 옮기기 쉽다.
- **대안:** NestJS + TypeORM, Express + Drizzle, SQLite 단일 파일 등.
- **트레이드오프:** 프레임워크 레일은 얇지만 설정·마이그레이션은 직접 챙겨야 한다.

---

## BSER API를 백엔드 proxy로만 처리

- **날짜:** 11일차
- **결정:** 공식 Open API는 브라우저에서 직접 호출하지 않고 백엔드만 호출한다.
- **이유:** 키 노출·CORS·rate limit 처리를 서버 한곳에서 할 수 있다.
- **대안:** 프론트에서 키 없는 공개 엔드포인트만 사용, BFF 없이 직접 호출 등.
- **트레이드오프:** 배포 시 프론트·백을 둘 다 올려야 하고, 레이턴시가 한 홉 더 생긴다.

---

## `EternalReturnClient` 인터페이스 도입

- **날짜:** 7일차
- **결정:** 플레이어·매치 데이터 접근은 인터페이스로 고정하고 mock/real 구현만 분리한다.
- **이유:** 실 API가 붙어도 페이지와 TanStack Query 계약을 바꾸지 않으려고.
- **대안:** `player.ts` 안에서 `if (mock)` 분기만 두기, 단일 axios 레이어 등.
- **트레이드오프:** 파일 수와 보일러플레이트가 늘고, DTO 조립용 메서드 하나를 더 뒀다.

---

## `X-User-Id` stub 인증

- **날짜:** 6일차
- **결정:** 실제 OAuth 전까지 헤더 `X-User-Id` 값을 신뢰하고 users 테이블에 stub으로 upsert한다.
- **이유:** 로그인 플로우 없이 favorites·search-history API를 먼저 검증하기 위해.
- **대안:** OAuth2, JWT 발급·검증, 세션 쿠키 등.
- **트레이드오프:** 배포 환경에서는 반드시 교체해야 하고, 지금은 보안 인증이 아니다.

---

## mock 데이터를 JSON으로 분리 (MSW 미사용)

- **날짜:** 4일차
- **결정:** mock은 `players.json`·`matches.json`을 loader가 읽는 방식으로 두고 MSW는 쓰지 않는다.
- **이유:** 의존성을 줄이고, 타입과 파일 구조를 바로 보면서 고치기 쉽게 하려고.
- **대안:** MSW, 서버 전용 mock 라우트, 인메모리만 두기 등.
- **트레이드오프:** 네트워크 레벨 인터셉트 검증은 안 하고, 로더 단위 테스트로만 본다.
